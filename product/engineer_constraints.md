# Технические ограничения — интервью с тех. лидом

**Интервью с:** Максим, Lead Backend Engineer, T-Pay Platform
**Дата:** Q4 2025
**Контекст:** Предпроектная discovery. Оценка реализуемости инициатив Q1 2026.

---

## 1. Упрощённый чекаут

**PM:** Можем ли мы убрать повторный ввод данных для заказов до 15 000 руб.?

**Максим:** Да, юридически согласовано. Но есть нюанс — весь процесс оформления сейчас завязан на legacy checkout service, которому 4 года. Он не умеет работать с pre-filled данными без валидации через отдельный шаг. Нам нужно либо написать обёртку, либо рефакторить точку входа. Оцениваю в **2 спринта**. Это реалистично для Q1, если стартуем в первых двух неделях января.

**PM:** Что может пойти не так?

**Максим:** Главный блокер — legacy checkout service не покрыт тестами на 60%. Любое изменение несёт риск регрессии. Нужна неделя на написание тестов перед основной работой — это уже учтено в оценке.

---

## 2. Движок персонализации

**PM:** Когда сможем использовать банковские данные для персонализации витрины?

**Максим:** Данные у нас есть, доступ разрешён для сегментов B и E. Проблема в инфраструктуре: data pipeline между банковским ядром и маркетплейсом работает в батч-режиме с задержкой 24 часа. Для персонализации нужен near-realtime. Оцениваю переход на стриминговую архитектуру в **4 спринта**. Это Q2, не Q1.

**PM:** Есть ли промежуточное решение на Q1?

**Максим:** Можем сделать «холодную» персонализацию — на основе вчерашних данных. Это 1 спринт, даст базовый эффект. Полноценный движок — только Q2.

---

## 3. A/B тестирование

**PM:** Можем ли мы запускать A/B тесты прямо сейчас?

**Максим:** На web — да, инфраструктура готова, **1 спринт** занял её запуск в прошлом квартале. На мобильном приложении — нет. Мобильный SDK для сплит-тестов в разработке, ожидаем готовности к февралю. Сейчас все тесты на мобайле — это по сути feature flags без статистической валидации.

**PM:** Это критично для нас?

**Максим:** Зависит от сегмента. Если тестируем A (молодёжь) — они в основном мобайл, результаты web-тестов нерелевантны. Нужно это учитывать при планировании экспериментов.

---

## 4. Push-уведомления

**PM:** Push для реактивации — что с инфраструктурой?

**Максим:** Технически доступно, уже используется в других продуктах банка. Но исторически CTR по маркетплейсу — **ниже 2%**. Это не техническая проблема, это поведенческая. Пуши работают у нас для транзакционных сообщений (статус заказа), но не для маркетинговых.

---

## 5. Реалтайм ценообразование от партнёров

**PM:** Партнёры хотят давать контекстные офферы в зависимости от профиля пользователя. Реализуемо в Q1?

**Максим:** Нет. У нас нет партнёрского API для этого сценария. Сейчас цены загружаются через фид раз в сутки. Разработка API на стороне партнёров — не наша зона, и дорожная карта на Q1 это не включает. Контекстные офферы через транзакционную историю (для E) — это отдельный сценарий, но он тоже упирается в data pipeline (см. пункт 2).

---

## 6. Категоризация товаров для МСБ

**PM:** Сегменту D нужны профессиональные категории — оптовые товары, документооборот. Когда?

**Максим:** Работа оценена в **3 спринта**: таксономия, маппинг существующего каталога, UI. Проблема — у нас нет ресурсов в Q1. Команда каталога занята миграцией данных до конца марта. Раньше Q2 не начнём.

**PM:** Есть ли обходной путь?

**Максим:** Быстрое решение — улучшить поиск под профессиональные запросы. Это 0.5 спринта и входит в Q1. Полноценные категории — нет.

---

## 7. Онбординг для пенсионеров

**PM:** Упрощённый онбординг для сегмента C — когда реалистично?

**Максим:** С технической стороны — **2 спринта**. Но у нас блокер на стороне дизайна: UX-исследование с сегментом C ещё не завершено, прототип не утверждён. Пока нет финального дизайна, мы не можем начать разработку. Если дизайн придёт в январе — успеваем в Q1. Если нет — Q2.
